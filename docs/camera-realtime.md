# 카메라 실시간 판독

브라우저 웹캠으로 X-ray 필름/인쇄물을 비추면, 주기적으로 프레임을 캡처해서 실시간으로 KL grade + Grad-CAM + 판독 소견을 화면에 표시.

## 구성

- `src/static/camera.html`: 웹캠 캡처 + 결과 오버레이 페이지 (순수 HTML/JS, 프레임워크 없음)
- `src/api.py`: `GET /camera` 라우트로 이 페이지를 서빙

## 동작 방식

1. 브라우저에서 `getUserMedia`로 웹캠 스트림 시작, 준비되면 "판독" 버튼 표시
2. 사용자가 "판독" 버튼을 누르면 그 시점의 프레임을 `<canvas>`에 그려서 PNG로 변환
3. 기존 `/predict` 엔드포인트에 그대로 전송 (Java를 거치지 않고 브라우저 → FastAPI 직접 호출 — 실시간성을 위해 왕복을 최소화하는 설계 선택)
4. 응답의 `gradcam_image_base64`를 비디오 위에 반투명으로 오버레이, 등급/신뢰도/소견을 함께 표시하고 "재개" 버튼으로 전환
5. grade 0~2는 초록, 3~4는 빨강으로 배지 색상 구분

원래는 2초 간격 자동 판독으로 만들었으나, 계속 재판독되는 게 불편하다는 피드백을 받고 **사용자가 버튼을 누른 시점에만 판독하는 방식**으로 변경 (아래 "실사용 중 발견한 문제" 4번 참고).

## 실행 방법

```powershell
uvicorn api:app --app-dir src --port 8000
```

브라우저에서 `http://localhost:8000/camera` 접속 (localhost는 HTTPS 없이도 카메라 접근이 허용되는 예외 주소).

## 검증

- 서버 라우팅(`/camera`, `/health`) 및 페이지 로드, 카메라 권한 거부 시 에러 처리는 확인 완료
- 실제 웹캠 촬영/추론은 개발 도구의 브라우저 샌드박스가 카메라 장치 접근을 차단해서 자동 테스트 불가 — 실제 브라우저(Chrome/Edge)에서 수동 확인 필요

## 실사용 중 발견한 문제 및 수정

실제 웹캠으로 테스트하다가 두 가지 문제를 발견해서 수정.

### 1. 등급과 리포트 내용이 서로 모순됨

퇴행성 관절염 X-ray를 화면에 띄워 웹캠으로 재촬영한 사진에서 화면 배지는 "Grade 3"인데, LLM이 생성한 리포트는 "확률 분포에서는 Grade 4가 80.9%로 가장 높아 실제로는 더 심할 수 있다"고 스스로 반박하는 내용을 적어서 나옴.

**원인 두 가지**
1. `confidence` 값을 `probs.max()`(전체 등급 중 최댓값)로 계산하고 있어서, 화면에 표시된 등급과 무관한 확률이 "신뢰도"로 표시되고 있었음 → `probs[predicted_grade]`(공식 등급 자신의 확률)로 수정
2. `generate_report()`에 등급별 확률 분포 전체를 LLM에 넘기고 있어서, LLM이 이걸 보고 "사실 다른 등급이 더 유력해 보인다"고 스스로 재해석하는 문장을 만들어냄 → 공식 등급과 그 등급의 신뢰도만 전달하고, 프롬프트에 "주어진 등급을 그대로 신뢰하고 반박하지 말라"는 지시 추가

### 2. X-ray가 아닌 이미지(예: 사람 얼굴)도 등급을 매김

모델은 무릎 X-ray로만 학습됐기 때문에 "이게 X-ray가 맞는지" 판단하는 기능이 아예 없음. 사람 얼굴을 비춰도 confidence 48%로 "Grade 1"이라는 그럴듯한 결과와 리포트를 만들어냄.

**해결**: 완벽한 판별 모델을 새로 학습시키는 대신, 저비용 휴리스틱 두 가지를 조합한 필터 추가 (`looks_like_xray()` in `src/api.py`)
- X-ray는 원본이 흑백이라 RGB로 저장돼도 R/G/B 채널 값이 거의 동일함 → 채널 간 평균 절댓값 차이가 임계값(12.0) 이상이면 컬러 이미지로 간주
- 모델 자체의 확신도가 낮으면(55% 미만) 학습 데이터와 동떨어진 입력일 가능성 높음
- 둘 중 하나라도 걸리면 등급 대신 "X-ray가 맞는지 확인해달라"는 안내 메시지로 대체하고, LLM 호출도 생략(비용 절감)

**한계**: 이건 완벽한 OOD(out-of-distribution) 탐지가 아니라 극단적인 오사용만 걸러내는 수준의 안전장치. 실제 서비스라면 별도의 "X-ray 여부 판별" 모델을 두는 게 정석.

**버그**: 위 필터 결과(`likely_xray`)가 numpy의 `bool_` 타입으로 계산되면서 FastAPI의 JSON 직렬화가 실패(`PydanticSerializationError`)하는 문제 발생 → `bool()`로 명시적 캐스팅해서 해결. numpy 연산 결과를 그대로 API 응답에 넣을 때는 항상 파이썬 기본 타입으로 변환해야 한다는 교훈.

### 3. 실사용 편의성 개선

- **1차 개선**: 유효한 X-ray 판독(`likely_xray: true`)이 한 번 나오면 2초 자동 스캔을 멈추고 "분석 완료" + "재판독" 버튼으로 전환
- **에러 메시지를 HTTP 상태 코드 대신 이해하기 쉬운 한국어 문장으로 변경**: `describeError()` 함수로 상태 코드(400/5xx)와 네트워크 실패를 구분해서 각각 다른 안내 문구 표시
- **API 명세서**: FastAPI가 `/docs`(Swagger UI)에 요청/응답 스키마를 자동 생성해주므로 별도 문서를 손으로 작성하지 않고, 각 엔드포인트에 `summary`/`description`을 추가해서 자동 문서의 품질만 높임

### 4. 자동 판독을 아예 없애고 수동 버튼 방식으로 재변경

1차 개선(자동 스캔 + 유효 결과 시 정지)도 결국 "준비 안 됐는데 계속 찍힌다"는 불편함이 남아있다는 피드백을 받고, **자동 판독 자체를 완전히 제거**. `setInterval` 기반 자동 스캔 로직을 지우고, 사용자가 "판독" 버튼을 누른 시점에만 프레임 1장을 캡처해서 `/predict`를 호출하는 방식으로 변경. 결과가 나오면 "재개" 버튼으로 바뀌고, 눌러야 다시 "판독" 버튼이 나타나는 명시적 2-상태(대기 ↔ 결과) 흐름.
